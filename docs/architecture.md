# Архитектура

Обзор технического устройства игры. Описание систем по отдельности — в
[current-systems.md](current-systems.md), план развития — в [v2-roadmap.md](v2-roadmap.md).

## Граф модулей

```
settings.py            (константы; ни от чего не зависит)
game_data/             (РЕЕСТРЫ — источник истины об игровых данных)
    registry.py        (универсальный Registry: key + числовой id)
    definitions.py     (BlockDef, ItemDef, Drop, FaceTextures)
    blocks.py          (реестр BLOCKS: 8 блоков, id 1-8)
    items.py           (реестр ITEMS: предметы-блоки + палки/еда/инструменты)
    mining.py          (break_time, can_harvest, get_drops)
    __init__.py        (validate(): перекрёстная проверка ссылок)
    ↑
rendering/
    texture_atlas.py   (атлас 16x16-тайлов, UV, иконки; фолбэк на цвета)
    ↑
    ├── inventory.py   (Slot, Inventory — чистые данные)
    │       ↑
    │   crafting.py    (рецепты + окно крафта; ursina UI)
    ├── world.py       (чанки, шум, меши, рейкаст; ursina Entity/Mesh)
    ├── player.py      (физика и камера; ursina)
    ├── ui.py          (HUD; ursina)
    └── game.py        (режимы, здоровье, голод; ursina Entity)
            ↑
        main.py        (точка входа: собирает всё, GameController)
```

Циклических импортов нет: `game_data` не знает о рендере и геймплее,
`rendering` знает только о `game_data`, остальные — о обоих.
Блоки и предметы — раздельные реестры: блок живёт в мире, предмет — в
инвентаре; связь через `ItemDef.placeable_block` и `Drop.item`.
Поведение (жидкость, коллизия, перекрываемость, еда, дропы, скорость
добычи) задаётся флагами определений — жёстких проверок ключей в коде нет.

## Ключевые решения

### Мир — словарь, а не сцена Ursina

Истина о мире — `World.blocks: dict[(x,y,z) -> block_id]`. Сущности Ursina —
только визуализация. Следствия:

- **Коллизии и рейкаст не используют физику Ursina.** AABB-коллизии игрока
  и выбор блока (алгоритм Аманатидеса—Ву) работают по словарю — O(1) на
  проверку блока, никаких коллайдеров.
- **Один меш на чанк.** 16×16 колонок собираются в одну `Mesh`
  (+ отдельная полупрозрачная для воды). Рисуются только грани, соседние
  с воздухом/прозрачным блоком. 49 чанков ≈ 122 тыс. вершин ≈ 61 тыс.
  треугольников — тривиально для GPU.
- **Перестройка чанка = пересоздание сущности.** Замена `entity.model` на
  живой сущности оставляла артефакты геометрии (проверено), поэтому
  `_apply_mesh` уничтожает сущность и создаёт новую (~4 мс на правку).

### Правки игрока отделены от генерации

`World.edits: dict[(x,y,z) -> id | None]` хранит все изменения игрока.
Базовый рельеф детерминирован (зерно + шум), поэтому выгруженный чанк
восстанавливается генерацией + наложением правок. Это же — будущая точка
подключения сохранений (миллстоун 11): сериализовать достаточно `edits`.

### Кадровый цикл

Ursina сама вызывает `update()`/`input()` у всех включённых сущностей.
Порядок между сущностями не гарантирован; наши системы не зависят от него:

| Сущность          | update() каждый кадр                          | input() |
|-------------------|-----------------------------------------------|---------|
| `Player`          | обзор, движение, гравитация, коллизии         | двойной пробел (полёт) |
| `GameController`  | подгрузка чанков, прицел, разрушение с ЛКМ    | вся остальная клавиатура/мышь |
| `GameState`       | тики голода/регенерации (только выживание)    | — |
| `CraftingUI`      | иконка на курсоре (только когда открыт)       | клики по слотам |

### Системы координат

- **Мир**: блок `(x, y, z)` — целые; занимает куб `[x, x+1) × [y, y+1) × [z, z+1)`.
  Y — вверх, дно мира y=0 (всегда камень), рельеф ≤ `WORLD_HEIGHT`.
- **Чанки**: `(cx, cz) = (floor(x/16), floor(z/16))` — только по горизонтали,
  чанк содержит все блоки колонок по вертикали.
- **Игрок**: позиция — точка у ног; AABB 0.6×1.8×0.6; камера на высоте глаз
  1.62 как дочерняя сущность (поворот по горизонтали — у игрока, наклон — у камеры).
- **UI** (`camera.ui`): x ∈ [-aspect/2, aspect/2], y ∈ [-0.5, 0.5].

## Особенности macOS (main.py)

1. **OpenGL 2.1**: шейдеры Ursina (GLSL 130/140) не компилируются →
   `Entity.default_shader = None`, рендер по классическому конвейеру.
   Вершинные цвета и туман (`scene.fog` → panda `Fog`) при этом работают.
2. **`aspectRatioChanged` не приходит при старте** → UI-линза остаётся
   немасштабированной → вызываем `window.update_aspect_ratio()` вручную.

На других платформах эти обходы не активируются (там работает
`unlit_with_fog_shader`, которому чанки задают `fog_start`/`fog_end`).

## Производительность (замерено, см. testing.md)

| Метрика                          | Значение (M-серия Mac, C-шум) |
|----------------------------------|-------------------------------|
| Средний FPS (vsync)              | 60.0                          |
| FPS в худших 5% кадров           | ~54                           |
| Генерация чанка                  | 1.1 мс медиана / 1.7 макс     |
| Построение меша чанка            | 2.4 мс медиана / 10.6 макс    |
| Правка блока (меш + соседи)      | 4.4 мс медиана / 6.3 макс     |
| Блоков в памяти (радиус 3)       | ~120 600                      |

Бюджет кадра 16.6 мс; подгрузка ограничена одним чанком за кадр
(генерация + меш ≈ 4–12 мс), поэтому просадки при беге редки и коротки.

## Известные ограничения

- Кэши `_height_cache`/`_coarse_cache` и `edits` растут неограниченно
  при дальних путешествиях (килобайты на чанк — заметно только в марафонах).
- `_generate_chunk` перебирает ВЕСЬ `edits` при генерации каждого чанка —
  O(все правки за игру). Сейчас незаметно, но деградирует после введения
  сохранений; лечится индексом правок по чанку (см. М11). `edits` также
  хранит правки, совпадающие с генерируемым рельефом (не отсеиваются).
- Перестройка меша всегда полная: обходятся все блоки чанка (включая
  полностью закопанные), каждый новый чанк дополнительно ремешит до 4
  соседей (чанки собираются по 2–3 раза при путешествии), а `set_block`
  на границе — до 3 синхронных ремешей. При текущих размерах это
  4–12 мс и не мешает; инкрементальная перестройка — М5.
- `_survival_slots` (инвентарь выживания на время креатива) живёт в
  `GameState`, вне `Inventory` — при сериализации инвентаря из креатива
  он был бы потерян. Учесть в М11.
- Вода статична: нет течения; блок под водой можно убрать — вода «висит».
- Проверка «в воде» — только по блоку на уровне ног.
- ~~Жёстко зашитые проверки id~~ — устранено в М1: поведение задаётся
  флагами реестров (`liquid`, `replaceable`, `food`, `drops`, ...).
- Пауза отсутствует: голод тикает, даже когда мышь отпущена по Esc.
- Смоук-проверки живут внутри main.py (`--smoke`), а не в tests/.
