# Решение по движку и стеку (ADR-001)

Статус: **принято**. Дата: 17.07.2026.
Контекст: переход от прототипа (Python + Ursina) к production-игре —
открытый воксельный мир, survival horror (см. [game_design_document.md](game_design_document.md)).

## Требования к стеку (из дизайна)

Хоррор этой игры живёт на **свете, тумане и звуке** — значит, движок
обязан давать production-качество именно там:

1. **Освещение**: динамические источники, мягкие тени, объёмный туман,
   глобальное освещение в пещерах и ночью. Это ядро атмосферы, не декор.
2. **Звук**: позиционное 3D-аудио, окклюзия, зоны реверберации, шины
   с эффектами, динамический микс. Звук — половина хоррора.
3. **Воксельный мир**: чанки, стриминг, меши (не сущность-на-блок),
   правки мира, пещеры. Дальность видимости **сознательно ограничена
   туманом** (дизайн-решение!), поэтому героический LOD не нужен.
4. **Производительность**: стабильные 60 FPS на средних машинах;
   симуляция мира (существа, «Просед», погода) в бюджете кадра.
5. **Итерации**: маленькая команда — скорость правок важнее максимума
   контроля. Редактор сцен, горячая перезагрузка, отладка.
6. **Кооператив в перспективе**: архитектура клиент/симуляция с самого
   начала, сетевой слой — потом.
7. **Дистрибуция**: сборки Windows/macOS/Linux без боли; Steam.
8. **Право и риск**: без лицензионных сюрпризов, желательно open source.

### Почему не остаёмся на Python/Ursina

Прототип доказал дизайн, но упёрся в потолки, которые нельзя чинить
точечно: классический конвейер OpenGL 2.1 на macOS (шейдеры вообще
отключены — объёмный туман и тени недостижимы), чистый Python в мешинге
и симуляции (уже сейчас 3–13 мс на чанк), примитивный звук, сборка и
дистрибуция Python-приложений. Для хоррора, который живёт на свете и
звуке, это дисквалификация, а не неудобство.

### Почему не Unity / Unreal / свой движок

- **Unity**: технически подходит, но закрытый код + лицензионная
  история 2023 года = стратегический риск для многолетнего проекта;
  преимущества перед Godot для нашего масштаба не компенсируют его.
- **Unreal**: избыточен для стилизованного воксельного мира; C++
  итерации и размер движка замедлят маленькую команду; воксели — чужая
  территория (дорогие плагины).
- **Свой движок (C++/Rust + wgpu)**: максимум контроля, но 1–2 года
  инфраструктуры до первого контента. Мы делаем игру, а не движок.

## Решение

**Godot 4.4+ (Forward+/Vulkan-Metal) + C# (.NET) для геймплея.**

- **Рендер**: Forward+, объёмный туман, динамические источники с
  тенями, декали, свой шейдерный стек поверх стандартного PBR.
  **Глобальное освещение: основной план БЕЗ полного GI.** VoxelGI
  исключён (требует запекания — непригоден для редактируемого мира);
  SDFGI — только эксперимент фазы P0 с явным критерием отказа
  (известные протечки света на кубической геометрии и цена обновления
  при правках мира). Художественная цель «ночь по-настоящему тёмная»
  достигается и без GI: солнце/луна + тени, вершинное затенение и AO
  из мешера (наследие прототипа), ручные источники, объёмный туман.
- **Воксели**: собственный чанковый мешер на C# — прямой порт
  проверенной архитектуры прототипа (словарь блоков → жадные меши
  чанков, границы, cross-растения). Видимость ограничена туманом
  (~120–160 м = радиус 8–10 чанков), поэтому C#-мешера достаточно;
  расчётный бюджет — 1–3 мс на чанк в фоновом потоке.
  **Запасной план**: модуль `godot_voxel` (Zylann) — зрелый C++
  воксельный стек для Godot (стриминг, LOD, сохранения), если C#-мешер
  не удержит бюджет. Честная оценка: это НЕ «дроп-ин» — у модуля
  собственная модель данных и стриминга, C++-граница с C#, кастомная
  сборка движка; переход означает замену слоя мира с сохранением
  реестров и симуляции (недели работы, не дни). Перед активацией
  зафиксировать лицензию модуля (MIT на момент написания — перепроверить)
  и условия сборки. Порог перехода — см. technical_roadmap.md.
- **Горячие места**: GDExtension (C++) допускается точечно (мешинг,
  шум), только после профилирования.
- **Данные**: реестры блоков/предметов/биомов/существ — данные
  (Godot Resources/JSON), экспортируемые из Python-реестров прототипа
  одноразовым скриптом. Идентификаторы и контракты сохраняются.
- **Звук**: позиционное аудио, шины с эффектами и реверб-зоны — из
  коробки. **Акустической окклюзии из коробки в Godot нет — это наша
  подсистема** (рейкасты к источникам + lowpass/затухание на шинах —
  стандартная практика сообщества; запасной путь — интеграция
  Steam Audio). На окклюзии держатся Слушень и Отголосок, поэтому она
  прототипируется рано (см. роадмап, P3–P4).
- **Шум**: FastNoiseLite (встроен в Godot, есть C#-версия) вместо
  pnoise2. Зёрна мира прототипа НЕ переносятся (другая реализация шума);
  переносится архитектура пайплайна и тесты детерминизма.
- **Симуляция**: фиксированный тик симуляции мира (10–20 Гц), отделённый
  от кадра, — тот же принцип, что в прототипе (данные отдельно от
  рендера); это же — будущая точка врезки кооператива (мир как
  авторитетная симуляция).

## Судьба прототипа

Прототип **сохраняется работоспособным** как:
1. живой дизайн-референс (все системы выживания в нём играбельны);
2. спецификация поведения: его тесты (детерминизм генерации, физика,
   инвентарь, крафт) — эталон при портировании;
3. песочница для быстрых экспериментов с генерацией (Python быстрее
   для проб формул).

Переносим: архитектуру (реестры-как-данные, чистый worldgen, меш на
чанк, правки поверх детерминированной генерации), алгоритмы, тесты-как-
спецификацию, процедурные текстуры как временный арт. Не переносим:
код построчно, зёрна миров, Ursina-специфику.

## Риски решения и меры

| Риск | Мера |
|------|------|
| C#-мешер не удержит бюджет | порог: >4 мс/чанк в фоне или >8 мс пик кадра → GDExtension C++; >этого → godot_voxel |
| SDFGI протекает на кубах / дорог при правках мира | это лишь эксперимент P0 с критерием отказа; основной план света от GI не зависит |
| Своя аудио-окклюзия окажется сложной | прототип окклюзии в P0-демо звука; запасной путь — Steam Audio |
| Godot C# на macOS/экспорт | проверяется в P0 (первый вертикальный срез собирается на все 3 ОС) |
| Соло-разработка утонет в контенте | процедурность + переиспользуемые «феномены» вместо ручных скриптов (см. GDD) |
| Импосибл-геометрия (нестандартные пространства) | отдельный прототип в P4; если дорого — вырезается без ущерба ядру |
